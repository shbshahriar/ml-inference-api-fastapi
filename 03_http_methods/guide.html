💡 The Solution

We build a REST API with FastAPI. Doctors interact with it through HTTP requests — adding new patients, retrieving records, updating details, or removing discharged patients. The data lives in a patients.json file for simplicity.

Core Concept

CRUD Operations

Every dynamic application performs these four actions — always.

CRUD → HTTP Mapping

CREATE → POST

Send data to the server to create a new resource

RETRIEVE → GET

Read / fetch data from the server

UPDATE → PUT

Modify an existing resource on the server

DELETE → DELETE

Remove a resource permanently from the server

CRUD in the Real World

📷

Instagram

  • C — Upload a new post
  • R — Scroll your feed
  • U — Edit caption
  • D — Delete a post
📊

Excel / Sheets

  • C — Add a new row
  • R — View the spreadsheet
  • U — Edit a cell value
  • D — Delete a row
🏥

Hospital System

  • C — Register new patient
  • R — View patient record
  • U — Update prescription
  • D — Discharge patient

Key insight: Dynamic applications — whether social media, productivity tools, or enterprise systems — always perform CRUD operations internally. HTTP methods are how we express those operations over the web.

💡 About {id}

The curly braces mark a path parameter. When a request arrives at /view/p001, FastAPI extracts p001 and passes it to your route function automatically.

How FastAPI Reads the File

main.py 
import json

# Load patients from JSON file
def load_data():
    with open("patients.json", "r") as f:
        return json.load(f)

# Save patients back to JSON file
def save_data(data):
    with open("patients.json", "w") as f:
        json.dump(data, f, indent=2)

⚠ Production Note

JSON file storage is for learning only. In a real hospital system you would use a proper database (PostgreSQL, MongoDB, etc). The concepts you learn here transfer directly.

HTTP Method

GET — Retrieve Data

Fetch data from the server. Safe, idempotent, no side effects.

GET /view — retrieve all patients
FastAPI route 
@app.get("/view")
def view_patients():
    data = load_data()
    return data
GET /view/{'{id}'} — retrieve one patient
FastAPI route 
@app.get("/view/{patient_id}")
def view_patient(patient_id: str):
    data = load_data()
    if patient_id not in data:
        raise HTTPException(status_code=404, detail="Patient not found")
    return data[patient_id]

Sample response for GET /view/p001

{
  "name": "John Doe",
  "age": 45,
  "city": "New York",
  "disease": "Diabetes"
}

ⓘ GET is Read-Only

GET requests must never modify data. They are safe (no side effects) and idempotent (same request always gives same result).

HTTP Method

POST — Create Data

Send data to the server to create a new resource. Not idempotent.

POST /create — add new patient
FastAPI route 
@app.post("/create", status_code=201)
def create_patient(patient_id: str, patient: Patient):
    data = load_data()
    if patient_id in data:
        raise HTTPException(status_code=400, detail="Patient already exists")
    data[patient_id] = patient.model_dump()
    save_data(data)
    return {"message": "Patient created"}

Request Body

{
  "name": "Alice Brown",
  "age": 28,
  "city": "Chicago",
  "gender": "female",
  "disease": "Asthma",
  "height": 160,
  "weight": 58
}

Response (201 Created)

{
  "message": "Patient created"
}

Status 201 signals that a new resource was created — distinct from 200 OK which means a generic success.

ⓘ POST is Not Idempotent

Calling POST twice with the same data creates two records (or an error if IDs collide). This distinguishes it from PUT and GET.

HTTP Method

PUT — Update Data

Replace or modify an existing resource. Idempotent — safe to repeat.

PUT /update/{'{id}'} — update patient record
FastAPI route 
@app.put("/update/{patient_id}")
def update_patient(patient_id: str, patient: Patient):
    data = load_data()
    if patient_id not in data:
        raise HTTPException(status_code=404, detail="Patient not found")
    data[patient_id] = patient.model_dump()
    save_data(data)
    return {"message": "Patient updated"}

Request Body (updated fields)

{
  "name": "John Doe",
  "age": 46,
  "city": "New York",
  "gender": "male",
  "disease": "Diabetes Type 2",
  "height": 175,
  "weight": 82
}

Response (200 OK)

{
  "message": "Patient updated"
}

PUT replaces the entire record. If you only want to update one field, that is PATCH — but we keep it simple here.

ⓘ PUT is Idempotent

Sending the same PUT request 10 times produces the same result as sending it once. The record always ends up in the exact same state.

HTTP Method

DELETE — Remove Data

Permanently remove a resource from the server. Idempotent but destructive.

DELETE /delete/{'{id}'} — remove patient record
FastAPI route 
@app.delete("/delete/{patient_id}")
def delete_patient(patient_id: str):
    data = load_data()
    if patient_id not in data:
        raise HTTPException(status_code=404, detail="Patient not found")
    del data[patient_id]
    save_data(data)
    return {"message": "Patient deleted"}

Request

DELETE /delete/p001

No request body needed — just the ID in the URL path.

Response (200 OK)

{
  "message": "Patient deleted"
}

⚠ DELETE is Permanent

Once a patient record is deleted from the JSON file, it cannot be recovered. In production systems you would use soft-delete (mark as inactive) rather than hard-delete.
